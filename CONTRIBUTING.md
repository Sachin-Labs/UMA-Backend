# Contributing to UMA Backend

Thank you for your interest in contributing to the UMA API! We appreciate your help in building a robust and secure attendance management system.

## 🚀 Simple Steps to Contribute

1.  **Fork** the repository to your own GitHub account.
2.  **Clone** your fork to your local machine.
3.  **Create a branch** for your changes (see naming conventions below).
4.  **Commit** your changes with a clear and descriptive message.
5.  **Push** your branch to your fork.
6.  **Open a Pull Request** (PR) against the main repository.

## 🏷 Branch Naming Conventions

Please prefix your branches to help us understand the context of your changes:
- `feature/` : New endpoints or core logic (e.g., `feature/slack-notification`)
- `fix/`     : Bug fixes or security patches (e.g., `fix/jwt-expiration`)
- `perf/`    : Performance optimizations (e.g., `perf/query-indexing`)
- `refactor/`: Code cleanup or restructuring

## 🛠 Types of Contributions
We welcome help with:
- **Backend Logic**: Improving API efficiency or adding new services.
- **Bug Fixes**: Fixing edge cases in attendance marking or leave logic.
- **Testing**: Adding unit tests (Jest) or integration tests.
- **Security**: Hardening authentication or data validation.
- **Documentation**: Improving API docstrings or READMEs.

## 📜 Basic Rules
- **Clean Code**: Follow established patterns (Service/Controller/Model) and linting rules.
- **Readable Commits**: Use descriptive messages like "Index organizationId in attendance schema" instead of "update db".
- **Documentation**: Update the relevant `README.md` or comments if you change environment variables or logic flow.
- **Security First**: Never commit private keys, real API secrets, or `.env` files.

Thank you for contributing to UMA! ❤️
